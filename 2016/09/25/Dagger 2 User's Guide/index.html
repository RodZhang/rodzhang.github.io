Dagger 2 User's Guide

发表于 2016-09-25   |   分类于 翻译   |     |  阅读次数

原文：http://google.github.io/dagger/users-guide.html

一个应用中最好的类是那些像 BarcodeDecoder、KoopaPhysicsEngine，AudioStreamer 这种。它们有依赖，可能是BarcodeCameraFinder、DefaultPhysicsEngine、HttpStreamer。

相反，最坏的类是那些占了空间但是实际上根本没有做多少事的，像 BarcodeDecoderFactory、CameraServiceLoader、MutableContextWrapper 这种。这些类只是笨拙的把一些有趣的东西捆绑在一起。

Dagger 是一个 那些 FactoryFactory 类的代替方案，它实现了依赖注入模式，你不用再承担编写那些模板的工作。它让你可以专注在那些有趣的类上。声明依赖，指定如何去满足它们。

通过建立在标准的 javax.inject 注解(JSR 330)上，每个类都是容易测试的。你不需要很多的模板仅仅是为了把 RpcCreditCardService 换成 FakeCreditCardService。

依赖注入不仅仅是为了测试。它同时可以方便的创建可复用，可互换的模块。你可以在你的所有app中共用同一个 AuthenticationModule。你还可以在开发期间运行 DevLoggingModule 而在生产环境中使用 ProdLoggingModule 使得在每种情形下得到正确的行为。

为什么Dagger 2是不同的

拥有各种各样的配置和注入API的依赖注入框架已经存在很多年了。因此，为什么要重复发明轮子呢？Dagger 2 是第一个以生成代码的方式实现了全杙的。它的指导原则是去模仿生成那些用户可能不得不手写来确保依赖注入简单，可追踪，高性能的代码。想了解更多背景，watch this talk (slides) by +Gregory Kick.

使用Dagger

我们将能通过构建一个咖啡机来展示依赖注入和 Dagger。关于你可以编译运行的完整代码，你可以看Dagger’s coffee example

声明依赖

Dagger 为你的应用构造实例并满足它们的依赖。它用 javax.inject.Inject 注解来识别哪个构造方法和域是它感兴趣的。

使用 @Inject 来注解 Dagger 应该用来创建实例的构造方法。当需要一个新的实例的时候，Dagger将获取需要的参数值并且调用这个构造方法。

Dagger 可以直接注入域(个人理解应该是成员变量)。在这个例子中，它获取了一个 Heater 实例 heater 和一个 Pump 实例 pump。

如果你的类有被 @Inject 注解的域，但是没有被 @Inject 注解的构建方法，Dagger 将注入那些需要的域，但是将不会创建新的实例。添加一个带 @Inject 注解的没有参数的构造方法来表明 Dagger 可能也会创建一个实例。

Dagger 也支持方法注入，尽管构造函数或字段注入通常是首选。

没有 @Inject 注解的类是不可以被 Dagger 创建的。

满足依赖关系

默认的，如上所述，Dagger 通过构造一个所请求类型的实例来满足每个依赖。当你需要一个咖啡机时，它将通过调用 new coffeeMaker() 同时设置它的可注入域来获取一个实例。

但是 @Inject 并不是在所有地方都有用：

  • 接口不能被构造
  • 第三方类不能被注解
  • 必须配置可配置的对象！

被 @Inject 和 @Provides 注解的类通过他们之间的依赖构成了对象的图。通过一组定义良好的根的图调用像应用的 main 方法或者一个安卓应用的入口代码。在 Dagger 2 中，这个集合是由一个包含带有不用参数并返回需要类型的方法的接口定义而成。通过用 @Component 注解这个接口并把 module 的类型作为 modules 的参数，然后 Dagger 2 会完整的实现这个约定。

它会生成一个和这个接口有相同名字并带有 Dagger 前缀的实现。通过调用这个实现的 builder() 方法来获取一个实例，同时用返回的这个 builder 来设置依赖并调用 build() 方法得到新的实例。

图中的绑定关系

上面的例子展示了如何去构造一个带有典型的绑定关系的 component，但是有多种机制可以用于绑定到图上。下面是可用的依赖，可以用来生成一个良好的 component：

  • Those declared by @Provides methods within a @Module referenced directly by @Component.modules or transitively via @Module.includes
  • Any type with an @Inject constructor that is unscoped or has a @Scope annotation that matches one of the component’s scopes
  • The component provision methods of the component dependencies
  • The component itself
  • Unqualified builders for any included subcomponent
  • Provider or Lazy wrappers for any of the above bindings
  • A Provider of a Lazy of any of the above bindings (e.g., Provider>)
  • A MembersInjector for any type

单例和作用域绑定

用 @Singleton 注解一个 @Provides 方法或者一个可注入的类。那么图将在所有使用它的地方都使用同一个实例。

一个可注入的类上的 @Singleton 注解也可以起到文档的作用。它提醒潜在的维护者，这类可以被多个线程共享。

由于 Dagger 2 在图中使用 component 实例来和 scoped 的实例关联，这些 component 自己需要声明它们想要表示的作用范围。例如，用 @Singleton 和 @RequestScoped 绑定同一个 component 没有任何意义，因为这些作用域有不同的生命周期，因此必须存在于 component 的不同生命周期中。要声明一个和给定作用域关联的 component，只需要简单的在 component 接口上添加这个作用域的注解。

Components 可能有多个作用域的注解。这说明它们都是同一个 scope 的别名，因此 component 可以包含其中任何一个。

可复用的 scope

有时，你想限制有 @Inject 注解的类的实例化次数或者 @Provides 注解的方法调用的次数，但是你不需要保证任何特定组件或子组件的生命周期内使用的是完全相同的实例。这在像 Android 这种分配内存的昂贵的环境中是非常有用的。

在这种情况下，你可以用 @Reusable。它不像其它的 scope 不和任何单独的 componet 关联，而是每个用这个绑定的 component 将会缓存它返回的或坏实例化的对象。

这意味着如果你在一个 component 中一个用 @Reusable 绑定的 module，但是只有一个 subcomponent 实际上用了这个绑定关系，那么只有这个 subcomponent 将会缓存这个绑定的对象。如果两个不是继承自同一个父类的 subcomponent 使用了这个绑定，那么它们都会缓存属于它们自己的对象。如果一个 component 的父类已经缓存了这个对象，那么它可以复用这个对象。

这里不保证 component 只调用一次绑定，因为使用 @Reusable 会返回一个可变的对象，或者是一个相同实例的引用，这是危险的。而在一个不可变的对象上使用 @Reusable 注解是安全的，如果你不在意它会分配多少次内存的话。

懒注入

有时你需要延迟实例化一个对象。对于任何绑定 T，你可以创建一个 Lazy，它将直到第一次调用 Lazy 的 get() 方法时才被实例化。如果 T 是一个单例的，那么 Lazy 在所有地方都是相同的一个实例。否则，每个注入点都将会得到一个属于它们自己的 Lazy 实例。无论如何，后续的在 lazy 的实例上的都会返回一个相同的 T 的实例。

Provider 注入

有时你需要返回多个实例而是仅仅一个。你有几个选择（Factories, Builders 等等），一个选择是注入一个 Provider 而不是只是一个 T。一个 Provider 在每次调用 .get() 方法时都会调用一次绑定逻辑。如果绑定逻辑是一个 @Inject 注解的构造方法，就会创建一个新的实例，但是 @Provides 注解的方法不保证这点。

注意：注入 Provider 可能导致代码混乱。通常你会想用一个工厂，或者一个 Lazy，或者重新组织你代码的使用期和结构，仅仅是注入一个 T。然而，@Provider 可以在某些情况下是很有用的。一种常见的用法是当你必须使用一个没有用你的对象的自然生命时间排队的传统架构(例如：servlets 是被设计为单例的，但只适用在请求特定数据的上下文)

限定符

有时单独的类型是不足以确定一个依赖的。例如，一个复杂的咖啡机应用可能想把加热器分割成水和加热板。

在这种情况下，我们添加一个限定符注解。它是一个本身就被 @Qualifier 注解的注解。这是 @Name 的声明，一个包含在 javax.inject 的限定符注解。

你可以创建你自己的限定符注解，或者直接使用 @Name。通过注解域或参数来应用它。类型和限定词注释将被用来确定依赖关系。

通过标相应的 @Provides 方法提供合格的值

依赖项可能没有多个限定符注解。

编译时代码生成

Dagger 2 的注解处理器可以生成像 CoffeeMaker_Factory.java or CoffeeMaker_MembersInjector.java 这样命名的源文件。这些文件是 Dagger 的详细实现。你不应该直接使用它们，尽管它们可以的通过注入分步调试。你仅仅应该引用那些以 Dagger 为前缀的 component。
